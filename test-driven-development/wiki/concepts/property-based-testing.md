Property-based testing generates random inputs to verify that invariants (properties) hold across all cases, rather than checking specific hand-written examples. It complements example-based TDD by exploring the input space far more broadly.

## Core Idea

Instead of writing:
```
test "addition is commutative":
    assert add(2, 3) == add(3, 2)
```

Write a property:
```
forAll(a, b):
    assert add(a, b) == add(b, a)
```

The framework generates hundreds or thousands of random (a, b) pairs and checks the property against each. When a failure is found, it shrinks the input to the minimal failing case.

## Key Concepts

### Properties (Invariants)
A property is a statement that should hold for all valid inputs:
- **Commutativity**: `f(a, b) == f(b, a)`
- **Associativity**: `f(f(a, b), c) == f(a, f(b, c))`
- **Idempotency**: `f(f(x)) == f(x)`
- **Roundtrip**: `deserialize(serialize(x)) == x`
- **Invariant preservation**: after any valid operation, the invariant still holds

### Generators
Custom generators produce valid random instances of your domain types. A `Money` generator might produce values with valid currencies and non-negative amounts.

### Shrinking
When a failing input is found, the framework systematically reduces it to the smallest failing case, making debugging tractable.

## Connection to TDD

Property-based testing complements rather than replaces TDD:

1. **TDD examples first** — write specific example-based tests during red-green-refactor to drive design
2. **Properties second** — once the API is stable, add property tests to explore edge cases the examples missed
3. **Properties catch what examples miss** — boundary conditions, overflow, empty collections, unicode strings

The TDD cycle remains example-driven because examples are concrete and guide interface design. Properties are added during the refactor phase or as a second pass to strengthen the test suite.

## Especially Powerful For

### [Value Objects](value-objects.md)
Value objects are ideal for property-based testing because they are immutable and have algebraic properties:
- Money addition is commutative and associative
- Creating a value object from its components and extracting them back yields the same components
- Equality is reflexive, symmetric, and transitive

### Pure Functions
Functions with no side effects are trivially property-testable. Given the same inputs, they always produce the same outputs, and properties can express the relationship between inputs and outputs.

### Serialization Roundtrips
Any serialize/deserialize pair should satisfy: `deserialize(serialize(x)) == x`. This catches encoding bugs, missing fields, and version incompatibilities.

### Parsers and Validators
For any valid input, `parse(format(x)) == x`. For invalid inputs, the parser should reject them gracefully.

### State Machines
Generate random sequences of valid operations and verify that invariants hold after each transition. This is model-based testing.

## Property-Based Testing vs. [Parameterized Tests](parameterized-tests.md)

| Parameterized Tests | Property-Based Tests |
|--------------------|---------------------|
| Fixed data chosen by developer | Random data generated by framework |
| Tests specific known cases | Explores unknown cases |
| Deterministic | Randomized (but reproducible via seed) |
| Good for requirements | Good for invariants |

Both reduce test duplication but serve different purposes. Use parameterized tests for known business rules and property-based tests for mathematical invariants.

## Limitations

- Properties can be hard to articulate for complex business logic
- Random inputs may not hit the specific cases that matter most (use example tests for those)
- Slow if generators are expensive or the property is computationally heavy
- Requires thinking about the domain at a more abstract level
