Mocking is the use of fake objects that record interactions for later verification. A powerful but potentially dangerous testing tool that must be used with design awareness.

## What Is a Mock?

A mock object:
1. Mimics the behavior of a real object in a controlled way
2. Records all interactions (method calls, arguments) while being used
3. Allows post-hoc verification: "did the code call the mock as expected?"

Mock objects are typically generated by a mocking library (Mockito for Java, unittest.mock for Python, Moq for C#, etc.). Most languages have at least one.

## Mock vs. Stub vs. Fake

| Term | Definition |
|------|-----------|
| **Fake** | A working implementation simplified for testing (e.g., in-memory database) |
| **Stub** | Returns canned responses to calls; no verification |
| **Mock** | Records interactions and allows verification after the fact |
| **Spy** | A real object that also records interactions |

Kent Beck's "Mock Object" pattern in [TDD by Example](../sources/tdd-by-example-kent-beck.md): create a fake version of an expensive/complex dependency to enable isolated testing.

## How to Use a Mock

```java
// 1. Create the mock
Display mockDisplay = mock(Display.class);

// 2. Pass it to the code under test via dependency injection
Calculator calc = new Calculator(mockDisplay);

// 3. Exercise the code under test
calc.calculate("1,2");

// 4. Verify the interaction happened as expected
verify(mockDisplay).show("1 + 2 = 3");
```

The critical step is **injection** — the mock must be passed in, not created inside the code under test. See [Dependency Injection](dependency-injection.md).

## When to Use Mocks

Use mocks to:
- Verify interactions with collaborating objects (not just outputs)
- Isolate the code under test from slow or unpredictable dependencies
- Test edge cases that are hard to trigger with real implementations
- Create [measurement points](dependency-injection.md) to observe code behavior

## Anti-Patterns and the Mockery

From [TDD Smells](tdd-smells.md): **The Mockery** — a test that uses so many mocks and stubs that the system under test isn't actually tested at all.

```java
// WRONG: mocking both the car AND the engine
Engine mockEngine = mock(Engine.class);
Car mockCar = mock(Car.class);
when(mockCar.getEngine()).thenReturn(mockEngine);
when(mockCar.start()).thenAnswer(/* ... */);
// There is no production code here. This is pure waste.
```

The Mockery is caused by:
- Excessive setup (poor separation of concerns)
- Tightly coupled design
- Testing implementations rather than behavior

**The correction:** review the design. Improve abstraction and separation of concerns. Practice test-first. Writing the test first prevents this because you'd never write a test this way from scratch.

## Mocks as Design Signals

If you need many mocks to test something simple, the design is telling you something:
- The unit under test has too many dependencies (low cohesion)
- Dependencies are too tightly coupled (hard to isolate)
- The interface boundaries are wrong

This is Farley's "listening to the code" — the [Excessive Setup](tdd-smells.md) and Mockery smells are design smell, not testing problems.

## Meszaros's Formal Distinction: State vs. Behavior Verification

[Gerard Meszaros](../entities/gerard-meszaros.md) formalized the distinction between two verification strategies in [xUnit Test Patterns](../sources/xunit-test-patterns-meszaros.md):

- **State Verification** — inspect the state of the [SUT](sut-and-doc.md) after exercising it and compare to expected state. Does not require mocks. "What is the result?"
- **Behavior Verification** — capture the indirect outputs (calls the SUT makes to its [DOCs](sut-and-doc.md)) and compare to expected behavior. Requires Mock Objects or Test Spies. "What did the SUT *do*?"

In Meszaros's taxonomy, a **Mock Object** is specifically a test double with pre-programmed expectations that verifies calls during or after exercise. A **Test Spy** records calls passively for later assertion by the test. Both support Behavior Verification but differ in when verification occurs:
- Mock: expectations set *before* exercise; verified automatically
- Spy: calls recorded *during* exercise; asserted *after* in the Verify phase

This distinction helps choose the right tool: use State Verification (simpler, less coupling) when possible; use Behavior Verification only when you need to verify indirect outputs that leave no observable state.

## GOOS: Mocks as a Design Tool (London School)

[Growing Object-Oriented Software](../sources/growing-oo-software-freeman-pryce.md) presents a fundamentally different view of mocking than "just isolating dependencies." In the [London school](london-school-tdd.md), mocks are the primary mechanism for **discovering and defining communication protocols** between objects.

### Interface Discovery
When implementing an object, you discover it needs a service. Rather than building the service first, you:
1. Define an interface for the service (from the client's perspective)
2. Mock that interface in the unit test
3. Write expectations for how the object will communicate with the mock
4. Later, implement the real service

The mock expectations become the **specification of the protocol** between objects.

### "Only Mock Types You Own"
Don't mock third-party APIs. Instead, write thin adapter layers that translate between your domain and external code. Mock your own adapter interfaces. This prevents:
- Tests coupled to poorly-understood external behavior
- Complex test setups to get third-party code into the right state
- Tests that break on library upgrades

### Don't Mock Concrete Classes
Mocking a concrete class leaves the relationship between objects implicit. Extracting an interface forces you to **name the role** the collaborator plays, making the design explicit and discoverable.

### Don't Mock Values
Values are immutable and have no identity. Just create instances and use them. If a value is too complex to construct, use a Builder pattern rather than mocking it.

### Listening to the Tests
Difficulty in mocking is design feedback:
- **Too many mocks** → object has too many responsibilities
- **Complex mock setup** → hidden dependencies or tight coupling
- **Can't mock without magic** → singleton/global dependency (extract and inject it)

> "When we find a feature that's difficult to test, we don't just ask ourselves how to test it, but also why is it difficult to test." — Freeman & Pryce

## Khorikov: Only Mock Unmanaged Dependencies

[Vladimir Khorikov](../entities/vladimir-khorikov.md) provides the most precise mocking guidelines in [Unit Testing: Principles, Practices, and Patterns](../sources/unit-testing-khorikov.md), grounded in his [four pillars framework](four-pillars-of-good-tests.md):

### The Core Rule

**Only mock unmanaged dependencies** — out-of-process dependencies whose side effects are visible to external systems (SMTP servers, message buses, third-party APIs). Never mock managed dependencies (your application's own database).

### The Reasoning

Khorikov distinguishes two types of communication:

- **Intra-system** (between classes inside the application) — these are **implementation details**. Mocking them couples tests to how objects collaborate, causing [false positives](test-fragility.md) on refactoring.
- **Inter-system** (between your application and external systems) — these are **observable behavior**. The communication pattern must be preserved for backward compatibility, making mocks legitimate here.

### Two Types of Out-of-Process Dependencies

- **Managed dependencies** (e.g., application database) — only your application accesses them. Their communication patterns are implementation details. Use real instances in integration tests.
- **Unmanaged dependencies** (e.g., SMTP, message bus) — other systems observe their side effects. Mock these to verify the communication contract is preserved.

### Mocks vs Stubs: A Strict Distinction

Khorikov simplifies Meszaros's five-type taxonomy into two categories:

- **Mocks** (mocks + spies) — emulate and examine **outcoming interactions** (commands that change dependency state)
- **Stubs** (stubs + dummies + fakes) — emulate **incoming interactions** (queries that return data)

**Never assert interactions with stubs.** A call from the SUT to a stub is an intermediate step, not an end result. Asserting it is overspecification — a common source of [test fragility](test-fragility.md).

### Connection to CQS

The mock/stub distinction maps to the Command Query Separation principle:
- Commands (side effects, no return value) -> Mocks
- Queries (return value, no side effects) -> Stubs
